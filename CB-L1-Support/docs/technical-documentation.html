1. System Overview

CB-L1 Support is an AI-powered Defect Intelligence Assistant designed for Honeywell's Connected Buildings L1 support team. It answers defect root-cause and fix questions by combining two knowledge sources:

  1. Historical Jira Defects — enriched, chunked, and indexed in a Qdrant vector database with hybrid retrieval (semantic + BM25 keyword + metadata scoring).
  2. Product Documentation — PDFs parsed via LlamaParse, semantically chunked, and indexed in a separate Qdrant collection with a full RAG pipeline.

Answers are streamed to the user in real-time via Server-Sent Events (SSE) with clear source attribution (defect KB vs. documentation) and include structured sections: root cause, fix steps, similar fixed defects, and optional documentation guidance.

Key design principle: The system is read-only to Jira (READ_ONLY_MODE = True). It never writes back to Jira, only reads. Sensitive data (emails, IPs, tokens) is automatically masked via PII filters.

6. LLM Provider Abstraction (llm_provider.py)

A unified API layer that makes all LLM calls provider-agnostic. A single LLM_PROVIDER setting switches the backend without changing any calling code.

On first use, _discover_models() queries the provider's /models endpoint to discover available model IDs. Results are cached via @lru_cache. If discovery fails (corporate proxy, auth issues), a hardcoded fallback catalog is used. Code-completion-only models are excluded automatically.

7. API Layer

7.1 Application Factory (api/app.py)

The FastAPI application is created with a lifespan context manager that:

  1. Warms the retriever singleton — loads the JSON knowledge base and connects to Qdrant once at boot (not per request).
  2. Checks RAG health — non-blocking reachability test for the documentation pipeline.
  3. Logs subsystem status — KB size, vector store readiness, LLM provider, RAG status.

The app also serves the pre-built React SPA from api/static/. A catch-all route serves index.html for all non-API paths (SPA client-side routing). The HTML shell is served with no-cache headers so UI changes are immediately visible.

CORS: Currently configured with allow_origins=["*"]. In production behind Render's single-origin architecture this is benign (API and SPA share the same origin), but should be tightened if the API is ever exposed separately.

9. Intent Router (chatbot/intent_router.py)

A deterministic-first classifier — no LLM call, pure rule-based. Classifies every question into one of four intents in strict priority order:

PriorityIntentTriggerExample
1 (highest)DEFECT_BY_KEYContains a Jira key (e.g. HCBS-95506)"What's the root cause of HCBS-95506?"
2SIMILAR_DEFECTSContains "similar defect", "like this", etc."Show similar fixed defects for login errors"
3DEFECT_DIAGNOSTICDiagnostic cue words (50+ patterns)"How to fix EOM publish failure?"
4GENERAL_HELPGreetings, capability questions (no tech terms)"What can you do?"

9.1 Diagnostic Cue Patterns (50+ entries)

Includes standard fix terms ("root cause", "how to fix", "workaround") AND symptom phrasings ("not showing", "missing", "blank", "stuck", "crash", "incorrect") so symptom reports are treated as defects, not generic questions.

9.2 Fallback Logic

If none of the above cues match:

  • Short input (≤2 words) without tech terms → GENERAL_HELP
  • Everything else → DEFECT_DIAGNOSTIC (the vast majority of questions are about defects)

10. Hybrid Retriever (chatbot/retriever.py)

The core retrieval engine. Combines four scoring signals over defect chunks stored in the Qdrant "defect" collection and aggregates scores to the defect level.

10.1 Scoring Formula

final = 0.45 × semantic + 0.25 × keyword + 0.15 × metadata + 0.10 × quality + 0.05 × fixed_boost
SignalWeightSourceDescription
Semantic0.45Qdrant vector search (cosine similarity)60 candidates, scores clamped to [0,1]
Keyword0.25Qdrant full-text index + in-memory BM2560 candidates, IDF-weighted term matching
Metadata0.15Query token overlap with structured fieldsComponents, labels, environment, fix_pattern, etc.
Quality0.10Pre-computed heuristicCompleteness score (root cause, fix, comments)
Fixed Boost0.05Resolution status+0.05 for fixed defects, −0.15 for cancelled

10.2 Retrieval Flow

  1. Embed query — Convert user question to 3072-dim vector via EmbeddingProvider.
  2. Qdrant semantic search — 60 nearest-neighbor candidates by cosine similarity.
  3. Qdrant keyword search — 60 candidates via full-text index (top 8 query tokens).
  4. Merge & deduplicate — Union by chunk_id; semantic hits retain vector score.
  5. BM25 re-scoring — Local IDF table (built at startup from KB) applied to keyword candidates.
  6. Defect-level aggregation — Group by issue_key, best chunk score wins.
  7. Relevance filtering — Drop defects below absolute floor (0.45) AND relative margin (0.75 × top score).
  8. Return top K — Sorted descending by final score.

10.3 Key Design Decisions

  • Qdrant full-text index — Replaces a local BM25 index file, so new defects added via the "Add defect" button are immediately searchable without a rebuild.
  • Incremental BM25 IDF — add_lexical_doc() folds new defects into the in-memory IDF table, avoiding staleness.
  • Relevance guard — Prevents "forced" answers from weakly-matched defects. Returns empty list when nothing relevant is found, letting the answer layer report honestly.
  • Exact key lookup — DEFECT_BY_KEY bypasses semantic search entirely; seeds similar defects from the target's own text.

11. Answer Generator (chatbot/answer_generator.py)

Two-path answer generation: LLM synthesis (primary) with a deterministic fallback.

11.1 LLM Path

Sends the user question + retrieved evidence to the complex-stage model (GPT-4o) with a grounding system prompt that enforces:

  • Evidence-only answers — no invented keys, counts, or dates.
  • Commit to a root cause (never say "not provided" or "unclear").
  • Cite actual Jira keys from the provided context.
  • Fixed response structure: Root Causes → Fix History → Suggested Steps → Similar Defects table.
  • Temperature 0.2, max_tokens 1800.

11.2 Deterministic Fallback

When the LLM is disabled or fails, a template-based answer is built from the retrieved metadata. Includes 12 fix-step templates mapped to the fix_pattern taxonomy:

Fix PatternTemplate Steps Summary
Config-EnvironmentVerify config → Compare envs → Update settings → Validate
Data CleanupIdentify bad records → Correct data → Rerun process → Add validation
Null HandlingReproduce null → Inspect upstream → Add guard/default → Regression test
Data Sync-IntegrationCheck sync status → Retry message → Remap entity → Validate E2E
Access-AuthorizationVerify role/token → Check stale identities → Update access → Retest
PerformanceReview logs → Identify slow step → Optimize → Re-measure
Job-Workflow-PipelineCheck scheduler → Rerun step → Validate output → Add monitoring
UI-FrontendReproduce → Check API → Fix component → Verify UI
Deployment/VersionConfirm version → Check release notes → Deploy/rollback → Validate
External DependencyConfirm origin → Engage owner → Apply workaround → Validate E2E
Logic CorrectionReproduce → Trace rule → Correct logic → Add test
UnknownTriage from similar defects → Reproduce → Apply closest fix → Escalate

11.3 Streaming

stream_diagnostic_answer() yields tokens incrementally via LLM streaming. The deterministic fallback emulates streaming by chunking its full text into 6-word groups.

11.4 Section Filtering

When the user selects specific sections via the options picker (e.g. only "root_cause"), _keep_sections() parses the LLM's markdown output and retains only the requested ## Heading blocks, discarding unwanted sections.

12. QA Orchestrator (chatbot/defect_qa.py)

Top-level orchestrator that ties intent routing, retrieval, synthesis, and documentation RAG fusion into a single flow. Provides both ask() (sync) and ask_stream() (SSE generator).

12.1 Two-Phase Interaction (Options Flow)

For defect-family questions, the system uses a two-phase flow:

  1. Phase 1 (no sections): Returns an options response with a section picker:
    • Root cause — Most likely cause(s) from similar defects
    • How to resolve it — Fix steps + documentation steps
    • Earlier similar defects — Table of fixed defects with Jira keys
  2. Phase 2 (with sections): Runs retrieval + synthesis only for the selected sections, reducing latency when the user wants only a subset.

12.2 Documentation RAG Fusion

When "resolve" is selected, the documentation RAG runs in parallel with defect answer synthesis (via a background ThreadPoolExecutor):

  1. build_doc_query() — LLM reframes the defect symptom into a neutral documentation question.
  2. ask_rag() — In-process RAG pipeline queries the doc Qdrant collection.
  3. Results are fused under labeled headers: "🗂️ From Historical Defects" and "📄 From Documentation".
  4. If the doc RAG returns a "not found" answer, the sources block is stripped to avoid misleading citations.

13. Prompt Engineering (chatbot/prompts.py)

13.1 System Prompt (DIAGNOSTIC_SYSTEM_PROMPT)

Key rules enforced:

  • Use ONLY provided context — never invent keys, counts, owners, or dates.
  • Prefer fixed/resolved defects when recommending fixes.
  • NEVER reply "root cause not provided" — always commit to the most likely cause, citing evidence.
  • If the similar defects section is empty, explicitly state "no matching defect found" and ask for more detail.
  • Required output structure: Root Causes → Fix History → Suggested Steps → Similar Defects table.

13.2 User Prompt Construction

build_user_prompt() assembles a retrieval-scoped payload:

  • Conversation history (if provided) — last 4 turns, truncated to 600 chars each, for reference resolution.
  • Question — the raw user question.
  • Current defect (if key lookup) — full metadata for the specific defect.
  • Retrieved similar defects — formatted evidence lines with key, status, relevance, root cause, fix.
  • Structure hint — scoped to the selected sections (root_cause / resolve) or full default.

14. Knowledge Base Build Pipeline

Run with python run_chatbot.py --build. Executes four stages in sequence:

Stage 1–4: build_knowledge_base.py

Loads three raw data files and normalizes them into an enriched knowledge base:

  • Input: raw_defects.json + issue_comments.json + llm_analysis.json
  • Process:
    • Normalize heterogeneous Jira/MCP field structures (top-level vs. nested fields).
    • Merge comments from the separate comments file.
    • Mask PII (emails, IPs, GUIDs, tokens, phone numbers).
    • Extract root cause: explicit Jira field → LLM analysis insight → resolution description → comment mining.
    • Extract fix applied: LLM analysis → resolution description → comment keyword search ("fixed", "resolved", "deployed").
    • Detect error signatures (HTTP 4xx/5xx, exceptions, timeouts, failures).
    • Classify fix_pattern (12 categories) via keyword scoring + LLM hints.
    • Classify defect_type (7 categories) via keyword scoring.
    • Compute quality_score (0–1 heuristic based on completeness).
    • Derive is_fixed and is_cancelled from status/resolution.
  • Output: defect_knowledge_base.json + .csv + synonym_dictionary.json

Stage 5: build_chunks.py

Each defect produces up to 5 typed chunks:

Chunk TypeContentPurpose
problem_chunkSummary + problem_text + symptom + error signatureMatch symptom descriptions
root_cause_chunkRoot cause + raw cause + key findingsMatch "why" questions
fix_chunkFix applied + workaround + resolution + solutionMatch "how to fix" questions
comment_insight_chunkKey findings + merged comment bodiesMatch discussion-based evidence
full_context_chunkAll metadata combinedBroad catch-all for hybrid queries

Stage 6: build_embeddings.py

Embeds all chunks via EmbeddingProvider (Azure OpenAI preferred, sentence-transformers fallback). Vectors are L2-normalized for cosine similarity. Outputs embeddings.npy + embedding_metadata.json.

Stage 7: migrate_defects_qdrant.py

Bulk upserts pre-computed chunks + embeddings into the Qdrant "defect" collection in batches of 200. Point IDs are deterministic SHA-256 hashes of chunk_id, so re-running is idempotent.

16. Chunking Strategy

Splitting each defect into 5 aspect-typed chunks is a deliberate design choice: a fix-oriented query like "how to fix EOM publish failure" will have a much higher cosine similarity to the fix_chunk of a relevant defect than to its problem_chunk, even when the problem text dominates the summary. Aspect-typed chunking prevents the problem text from drowning out the fix evidence.

Each chunk carries full defect metadata (status, resolution, priority, components, labels, fix_pattern, quality_score, is_fixed, is_cancelled) so the retriever can score metadata overlap without a separate lookup.

17. Embedding Pipeline

17.1 EmbeddingProvider

Unified embedding interface with automatic fallback:

  1. Remote (preferred) — Uses llm_provider.embed() which routes to Azure OpenAI or GitHub Models depending on EMBED_PROVIDER. Dimension: 3072 (text-embedding-3-large).
  2. Local (fallback) — sentence-transformers/all-MiniLM-L6-v2. Dimension: 384. Requires PyTorch. Only used when remote is unavailable.

All vectors are L2-normalized so inner-product search yields cosine similarity.

18. Defect Vector Store (chatbot/defect_vector_store.py)

Qdrant-backed store for the defect collection. Uses the same Qdrant cluster as the document RAG but a separate collection ("defect").

18.1 Indexes Created

Index TypeFields
KEYWORDissue_key, chunk_type, status, priority, resolution, fix_pattern, defect_type, components, labels, product, team, environment
BOOLis_fixed, is_cancelled
TEXT (full-text)text (word tokenizer, lowercase, min 2 / max 25 chars)

18.2 Idempotent Upserts

Point IDs are deterministic UUIDs derived from SHA-256 of chunk_id, so re-adding the same defect overwrites rather than duplicates. Before re-adding a defect, delete_by_issue_key() removes all existing chunks for that key.

18.3 Named Vector Detection

_detect_vector_name() handles Qdrant clusters that auto-assign a name to the default vector (e.g. "defect-vector") vs. those that leave it unnamed. This is detected once at startup so upsert/search calls address the correct field on any cluster.

19. On-Demand Defect Ingestion (chatbot/defect_ingest.py)

Allows adding new defects to the knowledge base at runtime without a full rebuild. Two modes:

19.1 Single Defect (add_defect(issue_key))

  1. Fetch issue + comments via Jira Cloud REST API v3.
  2. Convert ADF (Atlassian Document Format) to plain text.
  3. Normalize with the same _normalize_defect() pipeline as the bulk build.
  4. Chunk into 5 typed chunks.
  5. Embed via the shared EmbeddingProvider.
  6. Delete existing chunks for this key (idempotent update).
  7. Upsert new chunks into Qdrant.
  8. Update local KB in-memory + BM25 IDF table.

19.2 Bulk JQL (add_defects_by_jql(jql))

Same pipeline but runs a JQL search first (paginated via nextPageToken). Comments come embedded in fields.comment.comments instead of separate API calls. Capped at JQL_ADD_MAX_RESULTS defects per request. Both modes have SSE streaming variants for real-time progress.

20. Document RAG Pipeline

A self-contained in-process RAG pipeline extracted from a separate project. Answers "how to" and configuration questions from product documentation (PDFs). Uses a separate Qdrant collection ("rag_collection") on the same cluster.

Key design: The document RAG runs in-process (no separate HTTP server required). Originally it was queried over HTTP; it now shares the same Python process as the defect chatbot. All failures degrade gracefully — the defect answer is always returned, with or without the documentation section.

21. RAG Pipeline Architecture (rag_docs/pipeline.py)

Wires together four components as a lazy singleton:

VectorStore  →  AnswerGenerator  →  QueryEnhancer  →  RAGAgent

get_agent() builds the pipeline on first call and caches it. Construction validates that all required env vars are present for the selected provider.

21.1 Provider Selection

Controlled by RAG_LLM_PROVIDER and RAG_EMBED_PROVIDER independently, allowing chat on Copilot and embeddings on GitHub Models (Copilot lacks 3072-dim embeddings).

22. Document Ingestor (rag_docs/document_ingestor.py)

Stage 1: PDF → structured elements via LlamaParse (cloud API).

  • Parses each PDF page into Markdown, splitting into interleaved TEXT and TABLE elements.
  • Tracks section context (H1/H2/H3 headings) across pages.
  • Strips Markdown image references, preserving alt-text content.
  • Adds preceding/following text snippets (400 chars) for table context.
  • Handles corporate SSL proxy bypass internally.

23. Semantic Chunker (rag_docs/semantic_chunker.py)

Stage 2: elements → enriched, sequence-stamped chunks.

23.1 TEXT Chunking

  • Split on Markdown headings (#, ##, ###) via MarkdownHeaderTextSplitter.
  • Sections ≤ 3500 chars → single section_full chunk.
  • Larger sections → section_part chunks via RecursiveCharacterTextSplitter (1200 chars, 200 overlap).
  • Table-of-Contents entries and page headers are filtered out.

Every chunk carries: heading_h1, heading_h2, heading_h3, section_title, parent_path, sequence_id (global ordering).

26. RAG Client Bridge (chatbot/rag_client.py)

In-process bridge between the defect chatbot and the document RAG pipeline. Key features:

  • Query bridge: build_doc_query() uses a "cheap" LLM call to reframe a defect symptom into a neutral documentation question (removes "error", "bug", "fail" terminology that retrieves poorly against user guides).
  • Timeout safety: Runs under a ThreadPoolExecutor with a hard RAG_TIMEOUT (120s default). If exceeded, the defect answer is returned without the documentation section.
  • "No answer" checkpoint: Detects when the RAG says "does not specify/contain/mention" and strips the misleading sources block, replacing with a clean "not found" message.
  • Graceful degradation: All failures return None — the defect chatbot always works from its own evidence.

28. State Management (ChatProvider.tsx)

All chat state is managed via React Context in ChatProvider. Core features:

  • Multi-conversation support: Create, delete, rename, pin conversations. Each conversation has its own message history and context.
  • LocalStorage persistence: Conversations are debounce-saved (250ms) to localStorage so chat history survives page reloads.
  • Streaming engine: runStream() manages the SSE lifecycle — opening the connection, buffering tokens, handling options/sources/errors, and closing cleanly.
  • Options flow: When the backend returns an options event, the provider renders the section picker. When the user submits selections, submitOptions() re-sends the original question with the chosen sections.
  • Abort support: AbortController allows canceling in-flight streams.
  • Regenerate: Replays the original question + sections for any assistant message.
  • History turns: Last 8 {question, answer} turns are kept per conversation for backend context resolution.

30. SSE Streaming Implementation

A shared consumeSSE<T>() function reads the response body as a stream, buffering bytes and splitting on double-newline SSE frame separators. Each data: line is JSON-parsed and dispatched to the onEvent callback. Malformed frames are silently ignored (defensive coding for partial/corrupted network reads). 

// SSE consumption loop (simplified)
while (true) {
  const { value, done } = await reader.read();
  if (done) break;
  buffer += decoder.decode(value, { stream: true });
  // Split on "\n\n" (SSE frame separator)
  while ((sep = buffer.indexOf("\n\n")) !== -1) {
    const frame = buffer.slice(0, sep);
    buffer = buffer.slice(sep + 2);
    const dataLine = frame.split("\n").find(l => l.startsWith("data:"));
    onEvent(JSON.parse(dataLine.slice(5).trim()));
  }
}

31. Security Review

31.1 Strengths

  • Secrets externalized: All API keys/tokens loaded from .env via python-dotenv. Never hardcoded or committed.
  • Read-only mode: READ_ONLY_MODE = True is hardcoded — no Jira writes possible.
  • PII masking: Comprehensive regex-based masking of emails, phone numbers, IP addresses, GUIDs, URLs, Jira mentions, and long opaque tokens. Jira keys are explicitly preserved.
  • Input validation: Question length capped at 2000 chars. PDF uploads validated (extension, size, non-empty). JQL length capped at 2000 chars. Pydantic models enforce field constraints.
  • Parameterized queries: All Jira API calls use parameterized JSON bodies via httpx — no string concatenation for JQL.
  • Error handling: LLM/Qdrant/Jira failures degrade gracefully — never crash, always return a usable response. Internal error details are not leaked to users (generic Internal error message).
  • Deterministic UUID point IDs: SHA-256 content hashing prevents duplicate vectors in Qdrant.

31.2 Observations & Recommendations

  • CORS allow_origins=["*"]: Currently wide-open. Safe in the single-origin Render deployment but should be restricted if the API is ever exposed independently. Recommended: list specific allowed origins.
  • SSL verification disabled: verify=False on all httpx and QdrantClient connections. This is a deliberate workaround for the corporate SSL inspection proxy (Zscaler). Acceptable in the corporate environment but should be documented as a known security tradeoff.
  • ssl._create_default_https_context monkey-patch: pipeline.py globally disables SSL verification. This is necessary for the corporate proxy but should be removed when deploying outside the corporate network.
  • No rate limiting: The API has no request rate limiting beyond the LLM provider's own limits. Consider adding middleware rate limiting for production.
  • No authentication: All API endpoints are publicly accessible. Consider adding API key authentication for non-health endpoints.
  • File upload path traversal: The SPA catch-all route uses os.path.normpath + startswith(STATIC_DIR) to prevent path traversal — this is correctly implemented.
  • LLM prompt injection: The system prompt includes strong grounding rules ("use ONLY provided context") which mitigates casual prompt injection. However, adversarial inputs via the question field are not specifically filtered.

34. Code Quality Observations

34.1 Architecture Strengths

  • Separation of concerns: Clean layering — API routes → QA orchestrator → intent router → retriever → answer generator. Each module has a single responsibility.
  • Provider agnosticism: The llm_provider.py abstraction allows switching between Azure/GitHub/Copilot backends with a single env var. No calling code changes needed.
  • Graceful degradation: Every external dependency (LLM, Qdrant, Jira, RAG) has a fallback path. The system is always operational with some level of capability.
  • Deterministic fallback: When the LLM is disabled or fails, the 12-category fix-step template system still produces useful, structured answers from retrieved metadata alone.
  • Single-origin deployment: FastAPI serves both the API and the SPA from a single process, eliminating CORS complexity and proxy configuration.
  • Idempotent operations: Content-hash point IDs for both Qdrant collections ensure safe re-runs of ingestion pipelines.
  • Lazy initialization: Heavy components (LLM clients, Qdrant connections, embedding models) are loaded on first use and cached, so startup is fast and memory is allocated only when needed.
  • Incremental updates: On-demand defect ingestion and PDF ingestion allow extending the knowledge base at runtime without rebuilding.

34.2 Code Quality Notes

  • Typed dataclasses: DefectRecord and DefectChunk document the exact shape of data flowing between stages, with asdict() for JSON serialization.
  • Comprehensive PII masking: 8+ regex patterns covering emails, phones, IPs, GUIDs, URLs, Jira mentions, opaque tokens, and Markdown images. Jira keys are explicitly protected from being masked.
  • Well-documented modules: Every Python file has a detailed module docstring explaining purpose, behavior, and usage. Functions have type hints throughout.
  • Functional frontend state: ChatProvider uses functional state updates (callbacks with prev =>) to avoid stale closures — a common React pitfall handled correctly.
